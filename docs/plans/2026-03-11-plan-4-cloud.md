# Plan 4: Relix Cloud

> **For agentic workers:** REQUIRED: Use superpowers:subagent-driven-development (if subagents available) or superpowers:executing-plans to implement this plan. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Build the proprietary backend service handling auth, billing, push notifications, machine registry, and encrypted session history storage.

**Architecture:** Go HTTP API server with PostgreSQL for persistent data, Redis for sessions/caching. Stripe for billing. APNs/FCM for push notifications. Issues JWTs consumed by both relay and mobile app.

**Tech Stack:** Go 1.23+, net/http, PostgreSQL, Redis, Stripe Go SDK, firebase-admin (FCM), Apple APNs HTTP/2

**Spec:** `docs/specs/2026-03-11-relix-design.md`
**Depends on:** Plan 1 (protocol module)

## Task Outline

### Chunk 1: API Skeleton & Database (Tasks 1-5)

- [ ] **Task 1:** Init Go module, add dependencies
- [ ] **Task 2:** PostgreSQL migrations — users, machines, subscriptions, session_history tables
- [ ] **Task 3:** Database layer — users CRUD (create, get by ID, get by GitHub ID)
- [ ] **Task 4:** Database layer — machines CRUD (register, list by user, update status, delete)
- [ ] **Task 5:** HTTP router skeleton — health check, CORS, JSON error handling

### Chunk 2: Auth (Tasks 6-10)

- [ ] **Task 6:** GitHub OAuth — redirect to GitHub, handle callback, create/lookup user, issue JWT
- [ ] **Task 7:** Email auth — signup/login with email + password (bcrypt), email verification
- [ ] **Task 8:** JWT issuance — include user ID, tier, machine list in claims. 1h expiry, refresh token (30d)
- [ ] **Task 9:** Device code flow — generate code, poll endpoint, issue JWT on approval
- [ ] **Task 10:** Auth middleware — validate JWT, inject user into request context, check tier limits

### Chunk 3: Machine Registry & Pairing (Tasks 11-14)

- [ ] **Task 11:** Machine registration — agent calls POST /machines on first connect, gets machine_id
- [ ] **Task 12:** Machine list — GET /machines returns user's machines with last-seen status
- [ ] **Task 13:** Pairing code generation — POST /pairing/codes returns 6-digit code with 5min TTL (Redis)
- [ ] **Task 14:** Pairing code validation — POST /pairing/verify validates code, rate limits (5 attempts, 3s per IP)

### Chunk 4: Billing (Tasks 15-18)

- [ ] **Task 15:** Tier definitions — Free/Plus/Pro/Team limits (machines, sessions, history retention)
- [ ] **Task 16:** Stripe integration — create customer on signup, create checkout session for upgrades
- [ ] **Task 17:** Stripe webhooks — handle subscription.created, updated, deleted, payment_failed
- [ ] **Task 18:** Tier enforcement — middleware checks user tier against resource limits (machine count, session count)

### Chunk 5: Push Notifications (Tasks 19-21)

- [ ] **Task 19:** Push token registration — mobile app sends APNs/FCM device token, stored per user
- [ ] **Task 20:** APNs integration — send high-priority push for approval requests
- [ ] **Task 21:** FCM integration — same for Android, unified push interface

### Chunk 6: Session History (Tasks 22-24)

- [ ] **Task 22:** Encrypted blob storage — POST /sessions/:id/events stores E2E-encrypted event blobs
- [ ] **Task 23:** History retrieval — GET /sessions/:id/events returns encrypted blobs (mobile decrypts)
- [ ] **Task 24:** Retention enforcement — background job deletes history older than tier allows (7/30/90 days)

### Chunk 7: Deployment (Tasks 25-27)

- [ ] **Task 25:** Dockerfile — multi-stage build with migrations
- [ ] **Task 26:** Fly.io deployment config — fly.toml, PostgreSQL addon, Redis addon
- [ ] **Task 27:** Integration tests — full auth flow, machine registration, pairing, billing webhook
